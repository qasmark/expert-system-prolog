# Экспертная система поддержки платежной системы

## Что реализовано

- Дерево решений для 4 базовых кейсов:
  - `api_integration_error`
  - `anti_fraud_block`
  - `gateway_sbp_failure`
  - `user_payment_error`
- Отдельная стандартная база знаний в `knowledge/base_facts.pl`
- Отдельная обучаемая база знаний в `knowledge/learned_facts.pl`
- Дообучение дерева через `assertz`, если предложенный кейс не подошел
- Административный режим для просмотра и удаления дообученных кейсов
- Тесты `plunit`, включая проверки сохранения фактов через `assertz`

## Основные файлы

- `payment_support_expert_system.pl` — движок системы и интерфейс
- `knowledge/base_facts.pl` — стандартные вопросы, ветки, кейсы и признаки
- `knowledge/learned_facts.pl` — новые кейсы и признаки, добавленные в процессе дообучения
- `tests/payment_support_expert_system_tests.pl` — автоматические тесты

## Как запустить диагностику

Из корня проекта:

```powershell
swipl -q -s .\payment_support_expert_system.pl
```

После загрузки в консоли Prolog:

```prolog
?- run.
```

Система начнет задавать уточняющие вопросы и в конце выведет:

- наиболее вероятный кейс;
- объяснение, по какой ветке дерева он выбран;
- характерные признаки кейса из базы;
- рекомендуемые действия для оператора поддержки.

## Как работает дообучение

Если система предложила не тот кейс:

1. На вопрос подтверждения кейса нужно ответить `no`.
2. Система попросит ввести:
   - новый идентификатор кейса;
   - название кейса;
   - описание;
   - рекомендуемое действие;
   - новый различающий признак;
   - текст вопроса для этого признака;
   - должен ли ответ `yes` вести к новому кейсу.
3. После этого система:
   - добавит новую ветку дерева через `assertz`;
   - сохранит признак и кейс в `knowledge/learned_facts.pl`;
   - сможет использовать новый кейс уже в следующем запуске.

Идея такая: если оператор столкнулся с новой ситуацией, он не ломает базовую модель, а расширяет доменную зону новой веткой дерева.

## Как открыть административный режим

Сначала загрузить систему:

```powershell
swipl -q -s .\payment_support_expert_system.pl
```

Потом в консоли Prolog:

```prolog
?- admin_menu.
```

Доступные команды:

- `list` — показать все дообученные кейсы
- `delete` — удалить дообученный кейс
- `exit` — выйти из административного режима

При `list` выводится:

- идентификатор кейса;
- название кейса;
- родительский узел дерева;
- признак, который отличает новый кейс от старого

При `delete` система:

- спрашивает идентификатор кейса;
- удаляет связанную ветку дерева;
- убирает признаки и описание кейса;
- сохраняет обновленную обучаемую базу в `knowledge/learned_facts.pl`

## Как запускать тесты

```powershell
swipl -q -s .\tests\payment_support_expert_system_tests.pl -g run_tests,halt
```

## Что показывать на демонстрации

Удобный сценарий защиты:

1. Запустить `run/0` и показать базовую диагностику одного из 4 исходных кейсов.
2. Намеренно отклонить предложенный кейс и дообучить систему новым кейсом.
3. Повторно запустить `run/0` и показать, что новая ветка дерева уже используется.
4. Запустить `admin_menu/0`, показать `list`, затем удалить дообученный кейс через `delete`.
5. При необходимости прогнать тесты и показать, что механизм `assertz` и сохранение знаний покрыты автоматически.

## Ограничения текущей версии

- Это именно дерево решений, без Байеса и без фреймовой модели.
- Автоматическое добавление нового кейса реализовано для ситуации, когда нужно уточнить уже найденный лист дерева.
- Для полностью неизвестной ситуации без подходящего листа предусмотрено сообщение, но отдельный мастер построения новой корневой ветки пока не реализован.
